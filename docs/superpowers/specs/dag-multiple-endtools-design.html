三个方案对比：summary 节点如何接收多条入边（多个 endTools 同时汇聚）

总结
思路：summary 节点的 parentId 指向第一个 endTool；另外通过 endToolIds 字段记住所有 endTools。DAGCanvas 渲染边时，为每个 endTool 都画一条到 summary 的边（独立的 L 型折线）。
优点：无需引入虚拟汇聚节点，ReactFlow 原生支持，布局灵活；summary 的 parentId 依然有意义（链条串联不依赖此字段）
实现：DAGNode 类型新增 endToolIds?: string[]；边渲染时遍历 endToolIds 为每个 endTool 创建独立边

总结
思路：所有 endTool 水平边先向右延伸汇聚到同一 X 坐标（汇聚线），再垂直向下到 summary。
优点：视觉上呈现清晰的"扇出→汇聚"结构，特别适合并行工具多时。
实现注意：需通过 SVG 叠加层手动画汇聚折线，或 ReactFlow 的 addEdges + 虚拟边缘点；实现复杂度较高。

总结
思路：DAGNode 新增 additionalEdges?: { source: string; target: string }[] 字段，summary 节点通过此字段额外声明额外的入边。DAGCanvas 渲染时将这些额外边合并到 edges 数组中。
优点：原生节点结构不变，只需扩展边渲染逻辑。
缺点：DAGNode 混入边的概念，职责不纯；边不应该属于节点，应该独立管理。

推荐方案 A：多 EndTools 独立汇聚

summary 节点新增 endToolIds: string[] 字段记住所有 endTools；边渲染时，为每个 endTool 都画一条到 summary 的独立边。ReactFlow 原生支持，无需虚拟节点，修复 Bug 1（链条边误伤）也更自然。
